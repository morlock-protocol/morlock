# Morlock Protocol Specification
## Version 0.2

---

## Abstract

Morlock is an application-layer protocol enabling websites to expose structured, typed capabilities to autonomous AI agents. It defines a discovery mechanism, a command schema format, and a request/response wire format — all built on top of HTTP/JSON.

The goal: agents can interact with any Morlock-enabled site without vision models, DOM parsing, or browser automation.

### Changes from v0.1

Version 0.2 introduces mandatory security semantics that were absent or advisory in v0.1:

- **Command safety classification** (`read`, `write`, `unsafe`) drives auth and idempotency requirements.
- **Auth enforcement** — declared auth MUST be verified at runtime, not just advertised.
- **Idempotency keys** — write/unsafe commands require replay protection.
- **Rate-limit hardening** — `X-Forwarded-For` MUST NOT be trusted without explicit proxy configuration.
- **CORS defaults** — cross-origin access is denied by default, not allowed.
- **Error privacy** — unknown commands return generic 404, not names of valid commands.

---

## 1. Motivation

The web was designed for humans. AI agents attempting to use it face:

- **Unstructured HTML** with unpredictable layouts
- **No semantic capability declaration** (what can this site *do*?)
- **No typed interfaces** for commands or parameters
- **No standard auth pattern** for agent identity

Morlock addresses all four.

---

## 2. Definitions

- **Site** — an HTTP server implementing the Morlock protocol
- **Agent** — an autonomous software system consuming the Morlock protocol
- **Manifest** — the JSON document describing a site's Morlock capabilities
- **Command** — a named, typed, executable capability exposed by a site
- **Endpoint** — the single HTTP path handling all Morlock traffic

---

## 3. Discovery

### 3.1 Well-Known URI

Sites MUST serve their manifest at:

```
GET /.well-known/morlock
```

This follows [RFC 8615](https://www.rfc-editor.org/rfc/rfc8615) (Well-Known URIs).

Agents MUST check this path before attempting any other Morlock interaction.

### 3.2 Response Headers

Sites SHOULD include the following header on all responses:

```
X-Morlock: 0.2
```

### 3.3 CORS

Sites MUST respond to `OPTIONS` preflight requests with appropriate CORS headers.

Sites SHOULD restrict `Access-Control-Allow-Origin` to an explicit allowlist of trusted origins. The wildcard `*` is acceptable only for fully public, read-only, unauthenticated manifests. Sites exposing write or authenticated commands MUST NOT use `*`.

```
Access-Control-Allow-Origin: https://trusted-agent.example.com
Access-Control-Allow-Methods: GET, POST, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization, X-Api-Key, X-Morlock-Idempotency-Key, X-Morlock-Request-Id
Vary: Origin
```

If no CORS headers are returned, browsers will block cross-origin access. This is the safe default.

## 4. Manifest Format

### 4.1 Schema

```json
{
  "morlock": "0.2",
  "name": "Acme Store",
  "baseUrl": "https://acme.com",
  "endpoint": "https://acme.com/.well-known/morlock",
  "auth": {
    "type": "none | bearer | apikey | oauth2",
    "keyHeader": "X-API-Key",
    "tokenUrl": "https://acme.com/oauth/token",
    "scopes": ["read", "write"],
    "allowPublicRead": false
  },
  "rateLimit": {
    "requests": 60,
    "windowMs": 60000
  },
  "commands": {
    "search": {
      "description": "Search for products",
      "safety": "read",
      "params": {
        "q": {
          "type": "string",
          "description": "Search query",
          "required": true
        }
      },
      "returns": "Array of matching products"
    },
    "addToCart": {
      "description": "Add item to cart",
      "safety": "unsafe",
      "requiredScopes": ["cart:write"],
      "params": {
        "productId": { "type": "string", "required": true },
        "quantity": { "type": "number", "required": true }
      }
    },
    "setPreference": {
      "description": "Set a user preference",
      "safety": "write",
      "idempotent": true,
      "params": {
        "key": { "type": "string", "required": true },
        "value": { "type": "string", "required": true }
      }
    }
  },
  "contact": "api@acme.com",
  "tagline": "A no-nonsense API for serious developers",
  "voice": "Direct, technical, slightly dry",
  "agentName": "morlock/acme-store"
}
```

### 4.2 Required Fields

| Field | Type | Description |
|---|---|---|
| `morlock` | string | Protocol version. MUST be `"0.2"` |
| `name` | string | Human-readable site name |
| `baseUrl` | string | Canonical base URL of the site |
| `endpoint` | string | Absolute URL where commands are sent |
| `commands` | object | Map of command name → command schema |

### 4.3 Optional Manifest Fields

| Field | Type | Description |
|---|---|---|
| `auth` | object | Auth configuration (see 4.4) |
| `rateLimit` | object | Advisory rate limit info for agents |
| `contact` | string | Abuse/support email |
| `tagline` | string | One-liner that agents carry into conversations |
| `voice` | string | Tone/personality hint for agents rendering this site's responses |
| `agentName` | string | Registry name for this site's agent identity, e.g. `"morlock/acme-search"` |

### 4.4 Command Schema Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `description` | string | Yes | What the command does |
| `safety` | string | No | `"read"`, `"write"`, or `"unsafe"`. Default: `"unsafe"` |
| `params` | object | No | Map of param name → param schema |
| `returns` | string | No | Description of return value |
| `requiredScopes` | string[] | No | OAuth2 scopes required to call this command |
| `idempotent` | boolean | No | Whether the operation is naturally idempotent. Default: `false` |
| `examples` | array | No | Example request/response pairs |

### 4.4 Auth Config Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `type` | string | Yes | `"none"`, `"bearer"`, `"apikey"`, or `"oauth2"` |
| `keyHeader` | string | No | For `apikey`: which header the agent sends the key in |
| `tokenUrl` | string | No | For `oauth2`: token endpoint URL |
| `scopes` | string[] | No | Available OAuth2 scopes (informational, for agent discovery) |
| `allowPublicRead` | boolean | No | If `true`, commands with `safety: "read"` bypass auth. Default: `false` |

## 5. Command Execution

### 5.1 Request

```
POST {endpoint}
Content-Type: application/json
X-Morlock-Idempotency-Key: <key>   (required for write/unsafe commands)
```

```json
{
  "command": "commandName",
  "args": {
    "param1": "value1",
    "param2": 42
  },
  "requestId": "optional-uuid-for-tracing"
}
```

The `X-Morlock-Idempotency-Key` header is REQUIRED for commands with `safety: "write"` or `safety: "unsafe"` (see Section 7). Read commands do not require it.

### 5.2 Success Response

```
HTTP 200 OK
Content-Type: application/json
```

```json
{
  "ok": true,
  "requestId": "uuid",
  "result": {},
  "meta": {
    "executionMs": 42,
    "cached": false,
    "idempotentReplayed": false,
    "idempotencyKey": "key-if-provided"
  }
}
```

### 5.3 Error Response

```
HTTP 401 Unauthorized (or 403, 404, 409, 422, 429, 500 as appropriate)
Content-Type: application/json
```

```json
{
  "ok": false,
  "requestId": "uuid",
  "error": {
    "code": "AUTH_REQUIRED",
    "message": "Human-readable error description"
  }
}
```

### 5.4 Error Codes

| Code | HTTP Status | Description |
|---|---|---|
| `UNKNOWN_COMMAND` | 404 | Command not found. Message MUST NOT reveal valid command names. |
| `INVALID_PARAMS` | 422 | Missing required param or wrong type |
| `AUTH_REQUIRED` | 401 | Valid credentials not provided |
| `FORBIDDEN` | 403 | Credentials valid but insufficient permissions or scopes |
| `IDEMPOTENCY_KEY_REQUIRED` | 409 | Write/unsafe command sent without `X-Morlock-Idempotency-Key` |
| `RATE_LIMITED` | 429 | Too many requests |
| `COMMAND_FAILED` | 500 | Command handler threw an exception |
| `INTERNAL_ERROR` | 500 | Unexpected server error |

## 6. Command Safety

### 6.1 Classification

Every command has a `safety` field that classifies its side-effect behaviour:

| Safety | Side Effects | Auth Required | Idempotency Key Required | Description |
|---|---|---|---|---|
| `"read"` | None | Configurable | No | GET-equivalent. No state changes. |
| `"write"` | Yes, idempotent | Yes | Recommended | PUT/PATCH-equivalent. Repeating has same effect. |
| `"unsafe"` | Yes, not idempotent | Yes | Yes | POST/DELETE-equivalent. Repeating may cause duplicates. |

If `safety` is omitted from a command schema, it defaults to `"unsafe"`. This is deliberate — the safe default assumes the worst case.

### 6.2 Auth Bypass for Reads

When `auth.allowPublicRead` is `true` in the manifest, commands with `safety: "read"` bypass auth verification entirely. This allows sites to expose read-only discovery commands without credentials while protecting write operations.

When `auth.allowPublicRead` is `false` (the default), all commands require auth when `auth.type` is not `"none"`.

### 6.3 Scope Checking

Commands MAY declare `requiredScopes` — an array of OAuth2 scope strings. When present, the auth verifier SHOULD check that the presented token carries the required scopes. An empty array (or omitted field) means any authenticated caller is sufficient.

## 7. Idempotency

### 7.1 Problem

The protocol tells agents to retry on network errors (Section 9). Without deduplication, retries of write commands (`addToCart`, `createOrder`, `bookSlot`) may execute multiple times, causing duplicate side effects.

### 7.2 Mechanism

Agents MUST send an `X-Morlock-Idempotency-Key` header on all commands with `safety: "write"` or `safety: "unsafe"`. The key MUST be a unique, stable identifier for the logical operation (e.g., a UUID generated at intent creation time).

```
POST /.well-known/morlock
Content-Type: application/json
X-Morlock-Idempotency-Key: op_a1b2c3d4
```

**Key rules:**
- One key per logical operation
- Reuse the same key when retrying the same operation
- Generate a new key for a new operation

### 7.3 Server Behaviour

Sites MUST reject write/unsafe commands that arrive without an `X-Morlock-Idempotency-Key` header with `409 IDEMPOTENCY_KEY_REQUIRED`, unless the site has explicitly opted out of this requirement.

When a key is received:
1. If the key has not been seen before: execute the command, store the result keyed by the idempotency key.
2. If the key has been seen and a result exists: return the stored result with `meta.idempotentReplayed: true`. Do NOT re-execute the command.

Stored results SHOULD be retained for at least 24 hours. After the deduplication window expires, the key may be reused.

### 7.4 Naturally Idempotent Commands

Commands with `idempotent: true` (e.g., `setPreference(key, value)`) are semantically safe to repeat. Sites MAY relax the idempotency key requirement for these commands, but SHOULD still deduplicate when a key is provided.

## 8. Authentication

### 8.1 Enforcement

Auth declared in the manifest MUST be enforced at runtime. A site that declares `auth.type: "bearer"` MUST verify the token before executing any command (subject to `allowPublicRead` for read commands). A site MUST NOT start if auth is declared but no verification mechanism is configured. This is a startup error, not a runtime bypass.

### 8.2 None

No credentials required. `auth.type = "none"`.

### 8.3 API Key

```json
{ "type": "apikey", "keyHeader": "X-API-Key" }
```

Agent sends: `X-API-Key: sk-...`

### 8.4 Bearer Token

```json
{ "type": "bearer" }
```

Agent sends: `Authorization: Bearer <token>`

### 8.5 OAuth 2.0

```json
{
  "type": "oauth2",
  "tokenUrl": "https://site.com/oauth/token",
  "scopes": ["read:products", "write:cart"]
}
```

Agent obtains token via standard OAuth 2.0 client credentials flow. Per-command scope requirements are declared via `requiredScopes` in the command schema.

## 9. Rate Limiting

### 9.1 Manifest Declaration

Sites MAY declare rate limits in the manifest for agent awareness:

```json
{
  "rateLimit": {
    "requests": 60,
    "windowMs": 60000
  }
}
```

### 9.2 IP Resolution

Sites MUST NOT trust `X-Forwarded-For` headers without explicit knowledge of the proxy topology. The recommended approach:

- **No proxy** (direct connection): use the TCP socket IP. Ignore `X-Forwarded-For` entirely.
- **Known proxy count** (e.g., 1 for Cloudflare/Vercel): trust the rightmost N entries in `X-Forwarded-For`, use the next entry as the client IP.

Trusting leftmost `X-Forwarded-For` entries without validation allows attackers to spoof their IP and bypass rate limits.

### 9.3 Response Headers

When rate limiting is active, sites SHOULD include:

```
X-RateLimit-Remaining: 42
X-RateLimit-Reset: 1700000000
```

When the limit is exceeded, return `429` with error code `RATE_LIMITED`.

## 10. Agent Behaviour

Agents implementing Morlock SHOULD:

1. Check `/.well-known/morlock` before any other interaction
2. Cache manifests for the session (invalidate on 404 or schema change)
3. Validate param types client-side before sending requests
4. Respect `rateLimit` values declared in the manifest
5. Send `X-Morlock: 0.2` and a meaningful `User-Agent` header
6. Include `requestId` in all requests for tracing
7. Retry on network errors with exponential backoff (not on 4xx)

Agents MUST:

8. Send `X-Morlock-Idempotency-Key` on all write/unsafe commands
9. Reuse the same idempotency key when retrying the same operation
10. Generate a new idempotency key for each new logical operation

---

## 11. Versioning

The `morlock` field in the manifest declares the protocol version. Agents MUST check this field and warn or fail if the version is not supported.

Minor versions (0.x) may introduce additive changes. Major versions (1.0, 2.0) may introduce breaking changes.

A site serving `"morlock": "0.2"` is backward-compatible with agents that support v0.1 for read operations, but agents that do not send idempotency keys will receive `409` on write/unsafe commands.

---

## 12. Security Considerations

- Sites MUST NOT expose admin or destructive operations without authentication
- Sites MUST enforce auth declared in the manifest at runtime — not just advertise it
- Sites MUST NOT trust `X-Forwarded-For` for rate limiting without explicit proxy count configuration
- Sites MUST NOT default CORS to `Access-Control-Allow-Origin: *` on endpoints that handle write commands or authentication
- Sites MUST NOT reveal valid command names in error messages for unknown commands — return a generic 404
- Sites SHOULD rate limit by IP and/or API key
- Sites SHOULD validate all args server-side regardless of client-side validation
- Agents SHOULD NOT trust `result` data blindly — treat as untrusted input
- The protocol does not define agent identity; this is left to the auth layer

## 13. Relationship to Other Standards

| Standard | Relationship |
|---|---|
| `robots.txt` | Tells crawlers where not to go. Morlock tells agents what to do. |
| `sitemap.xml` | Describes page hierarchy. Morlock describes capabilities. |
| `llms.txt` | Static read-only context. Morlock is interactive. |
| OpenAPI / Swagger | Describes REST APIs. Morlock is simpler, agent-first, self-describing. |
| MCP | Local tool protocol. Morlock is web-native discovery. Complementary. |

## Appendix A: Minimal Implementation

A valid Morlock v0.2 implementation requires:

1. `GET /.well-known/morlock` returning a valid manifest with `"morlock": "0.2"`
2. `POST /.well-known/morlock` accepting `{ command, args }` and returning `{ ok, result }` or `{ ok, error }`
3. Auth enforcement when `auth.type` is not `"none"` — startup MUST fail if declared auth has no verifier
4. Rejection of write/unsafe commands without `X-Morlock-Idempotency-Key` (409)

Rate limiting, examples, and metadata remain optional.

## Appendix B: Migration from v0.1

| Change | Action Required |
|---|---|
| `safety` field defaults to `"unsafe"` | Add `safety: "read"` to read-only commands or they will require auth + idempotency keys |
| Auth enforcement | Provide a `verifier` function if `auth.type` is not `"none"`, or startup will throw |
| Idempotency keys | Agents must send `X-Morlock-Idempotency-Key` on write/unsafe commands |
| CORS defaults to `[]` | Explicitly set `corsOrigins` if cross-origin access is needed |
| Error messages are generic | Do not rely on error messages containing command names |
| `UNKNOWN_COMMAND` returns 404 | Previously returned 400 — update client error handling |
| `INVALID_PARAMS` returns 422 | Previously returned 400 — update client error handling |
